service discovery

  • use spring cloud: nacos
    • components:
      • provider app
      • consumer app
      • name server: use VIP(Virtual IP) or DNS for route
      • nacos server
      • nacos console
    • procedure
      1. provider app call nacos server‘s OpenAPI and register endpoint
      2. nacos server use heartbeat to detect the provider app status
      3. consumer app query for the instance list and pull instance list every 10s
      4. nacos server detect provider app abnormal and push update to consumer app by UDP
      5. comsumer app use Dubbo protocol to communicate with provider app

Annotations

  • @EnableDiscoveryClient
  • @LoadBalancedwill make RestTemplate instance configured for client-side load balancing
    • This annotation tells Spring to add an interceptor to the RestTemplate instance that will distribute requests across multiple instances of a service
  • @RequestBody BasicAuthDto dao indicate that the request body should be deserialized into a BasicAuthDto object
  • @Builder is a lombok annotation which help us to build a bean
    • usage:User user = User.builder().userId(dto.getUserId()).username(dto.getUserName()).build()
  • @Data: a lombok annotation which makes a bean a data class
  • @GenericGenerator: A Hibernate annotation that specifies the generator to use for primary key generation. In this case, it’s the UUIDGenerator provided by Hibernate
  • @Entity: A JPA annotation that marks the class as a JPA(Java Persistence API) entity, allowing it to be persisted to a relational database.
  • @Table: A JPA annotation that specifies the table name for the entity.
  • @Id: A JPA annotation that specifies the primary key field for the entity.
  • @Column: A JPA annotation that specifies the column name and properties for the annotated field.
  • @ElementCollection: A JPA annotation that specifies that the annotated field is a collection of simple values (i.e. not entities).
  • @CollectionTable: A JPA annotation that specifies the name of the table that stores the collection of simple values.
  • @PathVariable: URL query variable
  • @

Opentelemetry

  • Javaagent is suggested but we didn’t adopt.
  • Manual
  • Use openTelemetry.getTracer("instrumentation-library-name", "1.0.0") to get a tracer
  • Use tracer.spanBuilder("my span").startSpan() to start a span.
  • Use Span.end() to end a span.
  • Use Span.setParent(Context.current().with(parentSpan)) to set the parent of a span.
    • You can also omit the above step. Otl will automatically set the parent.
  • Each Context has its current Span.
    • Use Span.makeCurrent() to make the span the current span.
    • Use Span.current() to get the current span.
    • Use Span.fromContext(context) to get the current span of context.
  • Span can have attributes which are KV pairs.
    • Use Span.setAttribute("key", "value") to add an attribute.
  • Trace Semantic Convention: convert attribute from well-known protocol to span attribute.
    • Use Span.setAttribute(SemanticAttributes.HTTP_METHOD, "GET") to add a http attribute.
  • Span Event: structured log message on a Span.
    • Use Span.addEvent("Init") to add an event.
  • Span Link: associate the spans.
    • Use Span.addLink(parentSpan1.getSpanContext()) to add a link.
  • Span Status: Unset,Ok,Error
    • Use Span.setStatus(StatusCode.ERROR, "Opps") to set status.
    • Use Span.recordException(throwable) to record an exception.
  • Span Kind:
    • Client,Server,Internal,Producer,Consumer
      • Client&Server are synchoronous
      • Internal do not cross a process boundary
      • Producer&Consumer are asynchronous
    • Use Span.setSpandKind(SpandKind.CLIENT) to set a span kind.
  • ContextPropagation
    • provides a text-based approach to propagate context to remote services using the W3C Trace Context HTTP headers.
    1. Implement a TextMapSetter<HttpURLConnection> interface as setter
    2. Use openTelemetry.getPropagators().getTextMapPropagator().inject(Context.current(), transportLayer, setter)
    3. Implement a TextMapGetter<HttpURLConnection> interface as getter
    4. Use Context extractedContext = openTelemetry.getPropagators().getTextMapPropagator().extract(Context.current(), httpExchange, getter)
    5. Use extractedContext.makeCurrent() as parent context.

Fpllowing API are provided by opentelemetry-sdk

  • Sampler
    • There are four samplers:
      • AlwaysOnSampler which samples every trace regardless of upstream sampling decisions.
      • AlwaysOffSampler which doesn’t sample any trace, regardless of upstream sampling decisions.
      • ParentBased which uses the parent span to make sampling decisions, if present.
      • TraceIdRatioBased which samples a configurable percentage of traces, and additionally samples any trace that was sampled upstream.
    • Use SdkTracerProvider.setSampler(Sampler.TraceIdRatioBased(0.5)) to set sample rate
  • Span Processor
    • There are three processors:
      • SimpleSpanProcessor immediately forwards ended spans to the exporter
      • BatchSpanProcessor batches them and sends them in bulk
      • MultiSpanProcessor is multiprocess
    • Use SdkTracerProvicer.addSpanProcessor(BatchSpanProcessor.builer(grpcSpanExporter))
  • Exporter
    • There are five exporter:
      • InMemorySpanExporter: keeps the data in memory, useful for testing and debugging
      • Jaeger Exporter: prepares and sends the collected telemetry data to a Jaeger backend via gRPC. Varieties include JaegerGrpcSpanExporter and JaegerThriftSpanExporter.
      • ZipkinSpanExporter: prepares and sends the collected telemetry data to a Zipkin backend via the Zipkin APIs.
      • OpenTelemetry Protocol Exporter: sends the data in OTLP to the OpenTelemetry Collector or other OTLP receivers. Varieties include OtlpGrpcSpanExporter and OtlpHttpSpanExporter.

GRPC

  • procedure
    1. write a xxx.proto file in the same level as scr/main/java
    2. generate xxxProto.java, xxxReply.java, xxxRequest.java etc.
    3. generate xxxGrpc.java
    4. write a xxxImpl.java which extends xxxGrpc.xxxImplBase and override the callback
    5. Use ServerBuilder.forport().addService(new xxxImpl()).build().start() to obtain a rpc server
    6. Use ManagedChannelBuilder.forAddress(host, port).build() to get a rpc client
    7. Use xxxGrpc.newBlockingStub(channel) to get a stub
    8. Use xxxRequest.newBuilder().build() to build a rpc request
    9. Use stub.someMethod() to execute a rpc call
    10. Use channel.shutdown to shut down from client
    11. Use server.shutdown to shut down from server
  • others
    • There are 4 kinds of RPC
      • Unary: client–1–>server–1–>client
      • Server streaming: server holds a stream and write many responses
      • Client streaming: client holds a stream and write many request
      • Bidirectional streaming: both hold a stream and write many messages
    • There will be some methods correspond with rpc to be override in xxxImpl.java.
      These methods contain a parameter named StreamObserver<[Message class]>.
      This observer is passed from grpc framework, and we will use its callbacks for sending data.
      callbacks: onNext(), onError(), onComplete().
